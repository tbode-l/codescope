# 评分规则

总分 `100` 分，采用四维加权：

| 维度 | 权重 | 说明 |
|------|------|------|
| 关注度 | 25 | 主要看 `stars`，辅以 `forks`、`issues`、`last update` |
| 技术先进性 | 30 | 主要看 README 中体现的技术能力和项目新鲜度 |
| 配置门槛与产品形态 | 30 | 看安装步骤、依赖复杂度、是否开箱即用 |
| 生态开放性 | 15 | 看是否支持扩展、接入和二次开发 |

## 1. 关注度（25分）

- `stars` 为主，建议用 `log` 或分档归一化，避免头部项目碾压其他项目
- `forks` 作为热度补充
- `issues` 作为活跃互动信号
- `last update` 作为时间修正，越新越加分

## 2. 技术先进性（30分）

重点看 README 是否体现：

- 自动化和智能化程度
- 是否支持多模型 / 多仓库 / 多语言 / 多场景
- 是否具备索引、检索、Agent、RAG、插件化等能力
- 是否持续更新，且近期仍有维护

## 3. 配置门槛与产品形态（30分）

重点看：

- 是否一键运行
- 是否必须配置 API Key、环境变量、Docker、多服务
- 是 CLI、插件、Web、桌面端，还是纯库
- 是否能直接上手，而不是先折腾环境

## 4. 生态开放性（15分）

重点看：

- 是否开源、结构是否清晰
- 是否支持自定义模型 / provider / 插件
- 是否提供 API、CLI、SDK 或 hooks
- 文档、示例、扩展说明是否完整

## 建议实现方式

每个维度先算 0-100 的子分，再按权重求总分：

`总分 = 0.25 * 关注度 + 0.30 * 技术先进性 + 0.30 * 配置门槛与产品形态 + 0.15 * 生态开放性`

这样 `scorer.py` 里既好实现，也方便在论文里解释。
